# Internationalization (i18n) — Current State & Phased Plan

**Status: scaffolding only. The platform effectively serves pt-BR exclusively.**

Vitali advertises four languages (pt-BR / pt-PT / es / en) and ships the
request-time plumbing to select one, but there is **no usable translation
yet**. Real internationalization is a multi-sprint epic and is deliberately
deferred — Vitali is a Brazil-first product, and multi-country/multi-language
is a Phase 3 concern. This document is the honest record of what exists today
and the plan to get from here to actual multi-language support.

---

## 1. Current state (with file evidence)

### What IS wired (the scaffolding)

- **Settings** (`backend/vitali/settings/base.py`):
  - `USE_I18N = True`, `LANGUAGE_CODE = "pt-br"`
  - `LANGUAGES = [pt-br, pt-pt, es, en]` — four languages advertised
  - `LOCALE_PATHS = [BASE_DIR / "locale"]`
  - `TIME_ZONE = "America/Sao_Paulo"`
  - `django.middleware.locale.LocaleMiddleware` enabled
  - `apps.core.middleware.PreferredLanguageMiddleware` enabled — activates
    the user's stored language per request
- **Per-user preference**: `User.preferred_language` field
  (`backend/apps/core/models.py:466`).
- **Endpoint**: `GET / PATCH /api/v1/users/me/language/`
  (`backend/apps/core/urls.py`, `views.MeLanguageView`) — stores and reads the
  preference. Covered by `backend/apps/core/tests/test_i18n.py`.

### What is MISSING (why nothing is actually translated)

- **Empty catalogs**: `backend/locale/{pt_BR,pt_PT,es,en}/LC_MESSAGES/` contain
  only `.gitkeep` — **zero `.po`/`.mo` files**. There is nothing to translate
  *to*.

  ```
  $ find backend/locale \( -name '*.po' -o -name '*.mo' \) | wc -l
  0
  ```

- **Source strings are not marked**: across the entire backend there is exactly
  **one** `gettext` usage — a `gettext_lazy as _` import in
  `apps/core/admin.py`. Model `verbose_name`s, choices labels, DRF
  error/validation messages and admin text are all hardcoded pt-BR. Running
  `makemessages` today would yield a near-empty catalog.

- **Frontend has no i18n library**: `frontend/package.json` has no `next-intl`,
  `i18next`, or `react-intl`. All Next.js UI text is hardcoded pt-BR in JSX.

**Net effect:** selecting `es`, `pt-pt`, or `en` via the endpoint changes the
stored value and the activated Django locale, but every string the user sees is
still the hardcoded pt-BR source. There is no visible effect.

## 2. Phased plan

### Phase 1 — Backend string marking (convention + coverage)

Wrap all user-facing backend strings for translation and establish the
convention so new code follows it.

- Use `from django.utils.translation import gettext_lazy as _` for anything
  evaluated at import/module scope (model `verbose_name`, `help_text`, **choices
  labels**, field labels) — lazy so the string is resolved per request, not
  once at import time.
- Use `gettext` (eager) for strings built inside a request/view (DRF
  `ValidationError` messages, `APIException.default_detail`, serializer error
  messages, admin actions).
- Cover: `apps/*/models.py`, serializers, views, admin, and any
  user-facing constants.
- Outcome: source strings are extractable; English-source vs pt-BR-source
  decision documented (Vitali source strings are pt-BR — keep `LANGUAGE_CODE`
  as the source language and translate outward).

### Phase 2 — Catalogs (generate, translate, compile, CI-check)

- Generate: `python manage.py makemessages -l pt_PT -l es -l en`
  (pt_BR is the source language; no catalog needed unless source moves to
  English).
- Translate the resulting `.po` files (`backend/locale/<code>/LC_MESSAGES/django.po`).
  Start with pt-PT (cheapest delta from pt-BR), then es, then en.
- Compile: `python manage.py compilemessages`.
- **CI**: add a check that catalogs compile cleanly and that there are no
  untranslated/fuzzy entries above a threshold (fail the build on a broken
  `.po`). Commit `.po`; decide whether to commit `.mo` or compile at build time.

### Phase 3 — Frontend (Next.js)

- Add an i18n library (recommend **next-intl** for App Router locale routing).
- Extract hardcoded JSX strings into message catalogs
  (`messages/{pt-BR,pt-PT,es,en}.json`).
- Wire locale routing/switching and a language switcher in the UI.
- Align the frontend locale with the backend `preferred_language` (read it on
  login / from `/api/v1/users/me/language/`, persist the user's choice through
  the same endpoint), so the whole stack agrees on one language per user.

## 3. Gotchas

- **Lazy vs eager in module scope**: choices labels and `verbose_name` are
  evaluated at import time — use `gettext_lazy`, never `gettext`, or the string
  is frozen to whatever locale was active at import (i.e. always pt-BR).
- **Pluralization**: use `ngettext` / ICU plurals; pt and es plural rules differ
  from English. Don't concatenate counts into sentences.
- **Date/number/currency formats**: TZ is already `America/Sao_Paulo`; locale
  formatting (`USE_L10N` behavior, `django.utils.formats`, `Intl` on the
  frontend) must follow the active locale — pt-BR uses `,` decimal and `R$`,
  which must not leak into es/en views.
- **Do NOT translate domain/medical/regulatory terms**: TISS/TUSS codes, CID-10
  / ICD codes, FHIR resource names and field keys, CBO, ANS terminology, and
  drug/active-ingredient identifiers are identifiers, not prose — keep them
  verbatim. Translate only surrounding UI/help text.
- **Per-country regulation ≠ language**: pt-PT for Portugal and the eventual
  Angola (AO) rollout bring distinct regulatory regimes (data residency,
  billing). Language selection does not imply regulatory compliance — track
  those as separate Phase 3 work (see `docs/EPICS_AND_ROADMAP.md`).
